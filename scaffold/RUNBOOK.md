## Recommended capture workflow

**This runbook ships with a reference method for capturing real screenshots into `content/assets/`:**

1. Boot the app on an **iOS Simulator** (`xcrun simctl list devices booted` shows `(Booted)`).
2. An AI agent reads **this file** and drives **`maestro`** and **`xcrun simctl`** directly.
3. There is **no dedicated capture script** in the store-shots package — the runbook is the contract.

The engine never cares where the PNGs come from — only that files land under `content/assets/`
matching your `ctx.asset(...)` paths. Building for Android or the web, or capturing by hand?
Rewrite the capture sections below for your platform (e.g. Android emulator + Maestro, Playwright
for a web app) and keep the rest of the loop the same.

## The loop

0. **Read context** — link your project's sources of truth here, for example:
   - Product spec (e.g. `docs/product-overview.md`)
   - Marketing context (e.g. `.agents/product-marketing-context.md`)
   - Screenshot design guide / story / OCR checklist (e.g. `docs/aso/screenshot-guide.md`)

1. **Start preview**: run `bun run store:preview` in the background → `http://localhost:4317`
   - Editing `content/` triggers `--watch` reload. Re-navigate to see the latest.

2. **Overview**: open `/` with Playwright (or similar) and screenshot the gallery (save under `test-results/`)
   - Cross-check slide count, spec table, story, and OCR coverage against your design guide
   - Optionally run `WebSearch` for current ASO / competitor trends

3. **Per-slide review**: open `/slide/<id>` at full resolution for slides that need work

4. **Refine**: edit → hot reload → re-navigate (repeat 2–4)
   - Copy / colors / badges / structure → `content/config.ts`
   - Layout / design → `content/template.ts`

5. **Ship**: `bun run store:build` → `output/*.png` + `output/index.html` (matches what you saw in preview)

6. **Sync docs** — when copy or structure changes, update your design guide and `config.ts` together; update product spec if features changed

7. **Cleanup** (browser automation rules):
   - Stop the preview server
   - Delete review screenshots under `test-results/`
   - Close any headed browser you opened

## Things to remember

- Preview (`/slide/:id`) and final PNG (`output/:id.png`) share one render path — they always match.
- Missing assets render as placeholders; you can validate layout before art exists.
- Asset paths in `ctx.asset(...)` must match real files under `content/assets/`.

## Asset capture (the reference method)

Capture files into `content/assets/` from a booted simulator. The agent reads this section and operates `maestro` / `xcrun simctl` directly.

### Prerequisites

- App running on iOS Simulator (e.g. `bun run ios`). Confirm with `xcrun simctl list devices booted`.
- **Fill in:** any project-specific dev flags (ads off, screenshot mode env vars, Metro restart commands, etc.)

### Device & coordinate system

**Fill in for your project:**

| Item | Value |

| Simulator device | e.g. iPhone 17 Pro |

| `APP_ID` | e.g. `com.example.myapp` |
| Maestro `%` coords | relative to visible screen height in pt |

### Common pre / post processing

**Before each capture session** — lock status bar to 9:41:

```bash
xcrun simctl status_bar booted override --time 9:41 --batteryState charged --batteryLevel 100 --cellularBars 4 --dataNetwork wifi --wifiBars 3
```

**After the session:**

```bash
xcrun simctl status_bar booted clear
```

### How the agent works

- Identify elements with `maestro hierarchy` (JSON with `resource-id` and `bounds "[x1,y1][x2,y2]"`).
- Write temporary flows under `test-results/` and run:
  ```bash
  maestro test --debug-output test-results/maestro-debug test-results/<flow>.yaml
  ```
  (Keep all Maestro debug output under `test-results/`.)
- Capture screenshots directly into assets:
  ```bash
  xcrun simctl io booted screenshot content/assets/<relative-path>.png
  ```
- After each step, verify state via hierarchy or a screenshot before continuing. Re-run `maestro hierarchy` if elements are missing.

### Pitfalls

**Fill in** as you discover them. Common ones:

- Maestro does **not** interpolate `${}` inside swipe coordinates — use literal values.
- Swipe coordinates (% and point) can land non-linearly off-target — iterate with capture → visual check → adjust Y.
- After cold start, hierarchy may be empty briefly — retry with short sleeps.
- `inputText` does not switch keyboard language — configure simulator keyboards before capture if the keyboard must appear in-frame.

